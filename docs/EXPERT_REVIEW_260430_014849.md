## 2. CRITICAL 缺陷

### C-1: `writing_cb()` 忽略 `fl_write()` 返回值

- **文件**: `src/rocketdb_kvdb.c:1690`
- **严重性**: CRITICAL — 静默数据丢失
- **描述**: 在 KVDB init 阶段恢复 WRITING 记录时，Promoting WRITING→VALID 的 `fl_write()` 调用（写入 validity byte）的返回值被完全忽略。

```c
// rocketdb_kvdb.c:1687-1691
} else {
    /* Data intact → promote to VALID */
    uint8_t v = RDB_STATE_VALID;
    fl_write(db, ri->addr + 1, &v, 1);  // ← 返回值被忽略！
}
```

- **影响**: 如果 flash 写入失败（磨损块、电压不足），记录永久卡在 WRITING 状态。该记录在扫描时会被跳过，导致数据对用户不可见（静默丢失）。对比同文件第 1674 行的 `if (fl_read(...) != 0) {...}` 正确处理模式。
- **复现条件**: 在 init 扫描期间，存在 WRITING 记录且数据 CRC 匹配 → flash 第 2 字节（state）写入失败
- **建议修复**: 检查返回值，失败时至少记录 flash_errors 统计计数：
```c
if (fl_write(db, ri->addr + 1, &v, 1) != 0)
    db->stats.flash_errors++;
```

### C-2: `migrate_one` 提交失败导致活动扇区损坏

- **文件**: `src/rocketdb_kvdb.c:1222-1226`
- **严重性**: CRITICAL — NOR 约束违规
- **描述**: 如果 commit write（WRITING→VALID）在 line 1222 失败，函数返回 -1，但 `write_off` 在 line 1225 **未推进**。然而 flash 上已写入 header + key + value 数据。下次写入同一位置时，已写入的位（0 位）与新数据的 1 位冲突 → 违反 NOR 1→0 约束。

```c
// rocketdb_kvdb.c:1220-1226
/* Commit: WRITING → VALID */
uint8_t v = RDB_STATE_VALID;
if (fl_write(db, dst + 1, &v, 1) != 0)
    return -1;                         // ← 返回失败，但 header+data 已写入

db->sectors[db->active_sec].write_off += (uint16_t)rsz;  // ← 这行不会执行
return 0;
```

- **影响**: 下次写入到同一 `write_off` 偏移 → `sim_flash_write()` 检测到 0→1 违规 → 返回 -1 → 数据路径阻塞。在真实硬件上可能导致不可预测的 flash 控制器行为。
- **复现条件**: GC 迁移写入 header+key+value 成功，但 commit byte 写入失败（极罕见：部分页编程失败）
- **建议修复**: 失败时将 `write_off` 推进越过损坏区域，或标记该偏移为坏块跳过：
```c
if (fl_write(db, dst + 1, &v, 1) != 0) {
    db->sectors[db->active_sec].write_off += (uint16_t)rsz;  // 推进以跳过损坏数据
    return -1;
}
```

### C-3: Victim 扇区在迁移确认前被擦除

- **文件**: `src/rocketdb_kvdb.c:1330-1357 (gc_execute)`
- **严重性**: CRITICAL — 潜在数据丢失
- **描述**: `gc_execute()` 在 line 1343 调用 `scan_sector()` 遍历所有记录进行迁移。如果某个记录在被迁移的瞬间掉电，`migrate_one()` 返回错误且 `gc.err` 被设为 1（line 1296）。但函数在 line 1344 检查 `gc.err` 后返回 -1 → victim 尚未被擦除，数据安全。

然而问题出在：**迁移循环（line 1343）使用 `scan_sector()` 在一条记录迁移失败时立即停止**。之前已成功迁移并标记 DEAD 的记录已从 victim 中失效。如果此时函数因 err 返回 -1 而未擦除 victim，victim 中剩余的 VALID 记录仍然存在，但调用者能否正确处理？

实际检查后，`gc.err` 被正确设置且 victim 在 `gc.err != 0` 时不会被擦除。**此项目前的实现是正确的。**

**改判（重新审计后）**：此项不是 CRITICAL。`gc.err` 检查（line 1344）确保了迁移失败时 victim 不被擦除。降级为 MEDIUM — 但建议加注释说明此保护机制。

### C-4: `write_seq` 回绕到 0 (= RDB_SEQ_INVALID)

- **文件**: `src/rocketdb.h:247`, `src/rocketdb_kvdb.c:2037,2130`
- **严重性**: CRITICAL — 理论数据丢失
- **描述**: `write_seq` 是 uint32_t，从 1 开始递增（line 2037）。`RDB_SEQ_INVALID` 定义为 0（line 247）。当 write_seq 溢出归零时，所有新记录获得 seq=0，与 RDB_SEQ_INVALID 冲突。`RDB_SEQ_GT(seq, RDB_SEQ_INVALID)` 对 0 返回 false，导致记录被当作无效。

```c
// rocketdb.h:247
#define RDB_SEQ_INVALID     0u

// rocketdb_kvdb.c:2037
db->write_seq = 1;   // 从 1 开始

// rocketdb_kvdb.c:1187
db->write_seq++;      // 每次写递增，溢出回到 0
```

- **影响**: 理论风险 — uint32_t 需要约 42 亿次写入才溢出。对于典型嵌入式 NOR flash（10 万次擦除/扇区，16 扇区），需要 ~6.7 亿次写入，远超硬件寿命。**实际不可能触发**。
- **建议修复**: 将 `RDB_SEQ_INVALID` 改为 `0xFFFFFFFFu`，`write_seq` 从 0 开始：
```c
#define RDB_SEQ_INVALID     0xFFFFFFFFu
db->write_seq = 0;
```

### C-5: TSDB `ts_scan()` 在只读查询中修改 flash

- **文件**: `src/rocketdb_tsdb.c:533-548`
- **严重性**: CRITICAL — 查询产生意外磨损
- **描述**: `ts_scan()` 函数在扫描 TSDB 记录时，对于状态为 WRITING 的记录执行"就地恢复"：CRC 匹配则 promote 为 VALID，不匹配则 demote 为 DEAD。这个函数被 `rdb_tsdb_query()` 调用（只读查询），也被 `rdb_tsdb_init()` 调用（恢复扫描）。

```c
// rocketdb_tsdb.c:534-548
if (eff_state == RDB_STATE_WRITING) {
    uint16_t calc;
    uint32_t da = base + off + TS_REC_SZ;
    if (ts_data_crc(db, da, rh.data_len, &calc) == 0 &&
        calc == rh.data_crc) {
        uint8_t v = RDB_STATE_VALID;
        twr_f(db, base + off + 1, &v, 1);  // ← 写入 flash!
        eff_state = RDB_STATE_VALID;
    } else {
        uint8_t d = RDB_STATE_DEAD;
        twr_f(db, base + off + 1, &d, 1);  // ← 写入 flash!
        eff_state = RDB_STATE_DEAD;
    }
}
```

- **影响**:
  1. 只读查询意外消耗 flash 擦除寿命
  2. 查询中掉电可能导致记录状态不一致
  3. 在多读少量写入的场景，磨损被显著放大
- **复现条件**: 任何对包含 WRITING 记录扇区的查询（在 init 后立即查询未 seal 的 head 扇区即可触发）
- **建议修复**: 将 WRITING 恢复移到 init 阶段（独立恢复扫描），查询时仅跳过 WRITING 记录：
```c
if (eff_state == RDB_STATE_WRITING) {
    // 查询时仅跳过，不修改状态
    // init 阶段已通过独立扫描恢复了所有 WRITING 记录
    off += rsz;
    continue;
}
```

### C-6: `ts_seal()` 返回值在 `ts_rotate()` 中被丢弃

- **文件**: `src/rocketdb_tsdb.c:588`
- **严重性**: CRITICAL — 封存失败被静默忽略

```c
// rocketdb_tsdb.c:586-618
static rdb_err_t ts_rotate(rdb_tsdb_t* db) {
    /* Step 1: Seal the current head sector */
    ts_seal(db, db->head_sec, db->head_count, (uint16_t)db->head_off);
    // ← 返回值被忽略！
```

- **影响**: 如果 `ts_seal()` 失败（flash 写入错误），扇区头部保持未封存状态（count=0xFFFF, end_off=0xFFFF）。下次 init 时该扇区被当作 ACTIVE（未完成），但 head 已推进到新扇区 → 该扇区数据可能丢失或产生重复记录。
- **复现条件**: `ts_seal()` 内部 flash 写入失败
- **建议修复**:
```c
if (ts_seal(db, db->head_sec, db->head_count, (uint16_t)db->head_off) != 0)
    return RDB_ERR_FLASH;
```

### C-7: TSDB `head_off` / `end_off` 为 `uint16_t` — 限制扇区 ≤ 64KB

- **文件**: `src/rocketdb.h:727`, `src/rocketdb.h:509`
- **严重性**: CRITICAL — 大于 64KB 扇区时数据损坏

```c
// rocketdb.h:727
uint16_t head_off;       // Write frontier offset in head sector

// rocketdb.h:509
uint16_t end_off;        // Write frontier (0xFFFF = unsealed)
```

- **影响**: 使用 128KB 或 256KB 扇区时，超过 65535 的偏移值溢出。`head_off` 回绕到 0 后，新记录覆盖扇区开头的有效数据。`end_off` 溢出导致 init 无法定位正确的写前沿。
- **复现条件**: 扇区大小 > 64KB 且在单个扇区写入超过 64KB 数据
- **建议修复**: 将 `head_off` 和 `end_off` 改为 `uint32_t`（需注意结构体对齐和布局变化的影响）：
```c
uint32_t head_off;       // 支持最大 4GB 扇区
```

### C-8: TSDB 查询在 epoch 跨界的记录上可能缺失数据

- **文件**: `src/rocketdb_tsdb.c` (ts_scan, rdb_tsdb_query)
- **严重性**: CRITICAL — 查询结果不完整
- **描述**: TSDB 使用 epoch 机制管理时间戳 — 每个扇区有独立的 `time_base`，记录存储 `time_delta`（相对偏移）。当查询跨越 epoch 边界时，时间戳比较依赖于正确的 `time_base` 关联。如果边界扇区密封时 `time_base` 损坏或不正确，查询可能遗漏或重复记录。
- **影响**: 跨越扇区边界的查询返回不完整或重复数据
- **复现条件**: 查询时间范围跨越两个扇区，且第一个扇区的 time_base 与第二个不连续
- **建议修复**: 查询中添加 epoch 边界检查，确保每个扇区的 time_base 被正确应用于 delta 解码

### C-9: API — `strlen(key)` 在非空终止输入上的未定义行为

- **文件**: `src/rocketdb_kvdb.c:2087,2321,2398,2472`
- **严重性**: CRITICAL — 缓冲区越界读取

```c
// rocketdb_kvdb.c:2087
uint8_t kl = (uint8_t)strlen(key);   // key 必须是以 null 结尾的 C 字符串
```

- **影响**: 如果调用者传入非空终止的键，`strlen()` 会读取到 null 字符为止，可能越界访问，导致读取敏感内存区域或触发看门狗复位。
- **复现条件**: 恶意或错误输入（非空终止字符串）
- **建议修复**: 将 API 改为需要显式 `key_len` 参数。如果要保持向后兼容：
```c
rdb_err_t rdb_kvdb_set(rdb_kvdb_t* db, const char* key, uint8_t key_len,
    const void* val, uint16_t val_len);
```
或者至少在文档中明确要求 null 终止，并添加 `key[RDB_MAX_KEY_LEN] = '\0'` 安全复制。

### C-10: Simulator — Power-loss 从不产生部分写入

- **文件**: `test/sim/sim_flash.c:73-90`
- **严重性**: CRITICAL — 故障注入盲区

```c
// sim_flash.c:73-86 (验证阶段)
for (size_t i = 0; i < len; i++) {
    uint8_t old = f->mem[addr + i];
    uint8_t nw  = buf[i];
    if ((~old) & nw) return -1;   // 0→1 检查
    if (f->fault_ctx && fault_should_power_loss(f->fault_ctx, addr, (uint32_t)i)) {
        return -1;                 // ← 掉电在此触发！0 字节已写入
    }
}

// sim_flash.c:88-90 (写入阶段)
for (size_t i = 0; i < len; i++) {
    f->mem[addr + i] &= buf[i];   // ← 因掉电在验证阶段触发，此处永不到达
}
```

- **影响**: 掉电中断**始终在 0 字节写入后发生**。部分写入（写入一半后掉电）在故障注入测试中从未被测试。这意味着 KVDB/TSDB 对部分写入的恢复逻辑可能未经测试。
- **复现条件**: 每次 power-loss 注入
- **建议修复**: 合并验证和写入为单个循环：
```c
for (size_t i = 0; i < len; i++) {
    uint8_t old = f->mem[addr + i];
    uint8_t nw  = buf[i];
    if ((~old) & nw) return -1;
    f->mem[addr + i] &= nw;                          // 先写入
    if (f->fault_ctx && fault_should_power_loss(f->fault_ctx, addr, (uint32_t)i)) {
        return -1;                                    // 再检查掉电 → 部分写入
    }
}
```

### C-11: KVDB — `iter_gen` 在 GC 迁移后未递增

- **文件**: `src/rocketdb_kvdb.c` (gc_execute, 迭代器相关代码)
- **严重性**: CRITICAL — 迭代器返回过期/重复数据
- **描述**: KVDB 的迭代器使用 `iter_gen` 机制检测并发修改。当 GC 迁移记录时，记录从 victim 扇区移动到活动扇区。如果迭代器持有旧的 `gen` 值，它可能：
  1. 在旧扇区找不到记录（已迁移/标记 DEAD）
  2. 在新扇区看到重复记录（迁移后的新副本）
- **影响**: 迭代过程中 GC 触发 → 迭代器返回不一致数据
- **复现条件**: 迭代大数据库时 GC 并发运行
- **建议修复**: GC 完成后递增所有扇区的 `iter_gen`；迭代器检测 gen 变化后重新开始

## 3. HIGH 缺陷

### H-1: `mark_dead()` 失败留下两份 VALID 副本

- **文件**: `src/rocketdb_kvdb.c:1267-1269, 1282-1290`
- **严重性**: HIGH
- **描述**: 当 `mark_dead()` 失败时（line 1267），调用者仅递增 `flash_errors` 并继续。这导致 victim 扇区中的旧副本保持 VALID，而活动扇区中的新副本也是 VALID → 两份 VALID 副本。
- **影响**: `live_bytes` 统计膨胀；迭代器可能返回重复数据
- **建议修复**: 在 `mark_dead()` 失败时标记迁移为"未完成"，不在活动扇区 commit 新副本

### H-2: TSDB `rdb_tsdb_format()` 不强制最小扇区数

- **文件**: `src/rocketdb_tsdb.c` (rdb_tsdb_format)
- **严重性**: HIGH
- **描述**: TSDB 需要至少 2 个扇区来实现有意义的环形缓冲（head + tail）。但 `rdb_tsdb_format()` 未验证 `sector_cnt >= 2`。
- **影响**: sector_cnt=1 → 环形缓冲退化为单扇区，旋转时 head==tail 导致未定义行为
- **建议修复**: 在 `rdb_tsdb_format()` 中添加 `RDB_TS_MIN_SECTORS` 检查

### H-3: `ts_time_range()` 返回 `time_base` 而非实际最旧时间戳

- **文件**: `src/rocketdb_tsdb.c` (rdb_tsdb_time_range)
- **严重性**: HIGH
- **描述**: 时间范围函数返回扇区头部的 `time_base` 作为最旧时间戳，而非扫描实际记录找到最早时间。
- **影响**: 如果扇区第一个记录的时间戳与 time_base 不同，返回的 oldest 不准确
- **建议修复**: 扫描 tail 扇区找到实际的第一个有效记录时间戳

### H-4: `head_seq` 每 65536 次旋转回绕

- **文件**: `src/rocketdb.h:726`
- **严重性**: HIGH
- **描述**: `head_seq` 为 `uint16_t`，每 65536 次扇区旋转溢出。`RDB_SEQ16_GT()` 宏在差值 < 32768 时正确处理，但接近 32768 时判断可能错误。
- **影响**: 扇区排序在极长时间运行后可能出错（需 ~65536 次旋转，按 10 秒一次旋转约 7.5 天连续写入）
- **建议修复**: 改为 `uint32_t` 或增加溢出检测

### H-5: 7 个 API 函数缺少 `const` 修饰符

- **文件**: `src/rocketdb.h` (多个函数声明)
- **严重性**: HIGH — API 设计缺陷
- **描述**: 以下函数接受 `rdb_kvdb_t*` / `rdb_tsdb_t*` 参数但不修改数据库状态，未声明为 `const*`：
  - `rdb_kvdb_get()`, `rdb_kvdb_exists()`, `rdb_kvdb_count()`
  - `rdb_tsdb_get_latest()`, `rdb_tsdb_get_oldest()`, `rdb_tsdb_count()`, `rdb_tsdb_time_range()`
- **影响**: 调用者无法在 const 上下文使用；编译器不保护意外修改
- **建议修复**: 添加 const

### H-6: Stats 函数缺乏线程安全

- **文件**: `src/rocketdb.h` (rdb_kvdb_get_stats, rdb_kvdb_reset_stats, etc.)
- **严重性**: HIGH
- **描述**: 统计信息读取/重置函数无锁保护。在多线程环境中，并发读写 `stats` 结构体可能产生撕裂读取。
- **影响**: 并发环境下统计信息可能不正确
- **建议修复**: 在 stats get/reset 函数内部加锁

### H-7: `rdb_kvdb_exists()` 返回 `int` 而非 `rdb_err_t`

- **文件**: `src/rocketdb.h`
- **严重性**: HIGH — API 不一致
- **描述**: `rdb_kvdb_exists()` 返回 `int`（RDB_TRUE/RDB_FALSE），而其他所有函数返回 `rdb_err_t`（RDB_OK/RDB_ERR_*）。
- **影响**: 调用者需要记住特殊规则；可能在 switch-case 中遗漏处理
- **建议修复**: 返回 `rdb_err_t`，用 RDB_OK/RDB_ERR_NOT_FOUND 表达存在性

### H-8: TSDB 大记录写入路径依赖 `head_off` 精度

- **文件**: `src/rocketdb_tsdb.c`
- **严重性**: HIGH — 级联故障
- **描述**: 大记录（超过 RDB_STACK_BUF_SIZE）的写入分多步完成。如果 `head_off` 因之前的错误（如 C-7）已经损坏，大记录写入可能覆盖有效数据。
- **影响**: 一个错误导致级联数据损坏
- **建议修复**: 大记录写入前验证 `head_off` 对齐和范围

### H-9: TSDB `head_count` 在旋转后不反映实际 count

- **文件**: `src/rocketdb_tsdb.c:622`
- **严重性**: HIGH
- **描述**: `ts_rotate()` 在 line 622 设置 `db->head_count = 0`，但未验证密封扇区的 count 是否正确写入。
- **影响**: 如果 `ts_seal()` 失败（且返回值被忽略，见 C-6），head_count=0 但扇区头部 count 仍为 0xFFFF
- **建议修复**: 修复 C-6 后此问题自动缓解

### H-10: `space_info` / `wear_info` 无锁读取元数据

- **文件**: `src/rocketdb_kvdb.c` (rdb_kvdb_space_info, rdb_kvdb_wear_info)
- **严重性**: HIGH
- **描述**: 读取扇区元数据（status, erase_cnt, garbage_bytes）时不加锁。同时的写入/GC可能修改这些字段。
- **影响**: 返回不一致的空间/磨损信息
- **建议修复**: 函数内部加锁

### H-11: `rdb_tsdb_get_latest/oldest` 输出缓冲区不足时静默截断

- **文件**: `src/rocketdb_tsdb.c`
- **严重性**: HIGH
- **描述**: 如果用户提供的缓冲区小于记录的数据长度，数据被截断但无错误指示。
- **影响**: 调用者收到部分数据而不知情
- **建议修复**: 返回 RDB_ERR_TOO_LARGE 或设置 out_len 为实际需要的大小

### H-12: 硬编码 CRC 字节数无 `_Static_assert` 保护

- **文件**: `src/rocketdb_tsdb.c:467` (ts_seal — CRC over 18 bytes)
- **严重性**: HIGH
- **描述**: `ts_seal()` 对 header 前 18 字节计算 CRC：`rdb_crc16(&h, 18)`。如果 `rdb_ts_sector_hdr_t` 结构体因新增字段而扩展，CRC 覆盖范围错误但编译无警告。
- **影响**: 结构体变更后 CRC 验证不正确
- **建议修复**: 添加编译时断言：
```c
_Static_assert(sizeof(rdb_ts_sector_hdr_t) >= 20, "TS sector hdr size changed");
```

### H-13: GC 迁移循环缺乏超时保护

- **文件**: `src/rocketdb_kvdb.c:1343 (scan_sector)`
- **严重性**: HIGH
- **描述**: `gc_migrate_cb` 对每个记录执行 flash 读写。如果 victim 包含大量记录，GC 可能长时间阻塞。无限循环无超时保护。
- **影响**: 实时系统中长时间阻塞（数百ms到数秒）
- **建议修复**: 添加 yield 调用或最大迁移记录数限制

## 4. MEDIUM 改进项

### M-1: `RDB_ALIGN_UP` 宏对零输入行为需要文档化
- 当前实现 `(x + (g - 1)) & ~(g - 1)` 当 g=0 时除零

### M-2: TSDB 最大记录大小未考虑扇区头部
- `max_data_len` 计算应显式减去 sector header 和 data_start

### M-3: GC 后统计信息不归零
- 格式化后某些 stats 字段（如 gc_runs）保留旧值

### M-4: 两个引擎的错误恢复路径部分缺失
- KVDB: 活动扇区损坏时无自动恢复
- TSDB: 所有扇区密封时无新写入路径

### M-5: KVDB 元数据缓存在 erase 后未失效
- `sector->create_seq` 重置为 0，但其他元数据字段保留旧值（write_off, garbage_bytes 已重置）

### M-6: TSDB sector 头部恢复: count/end_off=0xFFFF 的扇区
- 未封存扇区在 init 时被当作 ACTIVE，但如果 CRC 不匹配则当作 CORRUPT 擦除，可能丢失数据

### M-7: `find_latest()` 对 `RDB_MAX_KEY_LEN` 键的性能退化
- 键越长，线性扫描越慢

### M-8: `ts_corrupt_skip()` 跳过策略可能漏过有效记录
- 当 magic byte 损坏时跳过固定大小，可能跳过嵌入式有效数据

### M-9: `rdb_dedup_set_t` 在大型 GC 时的内存限制
- 去重集合固定大小 16 条目，大扇区可能无法全跟踪

### M-10: sim_flash 写粒度验证信息不足
- 对齐错误时返回 -1 但无日志记录违规细节

### 🟡 C-8: TSDB epoch 查询边界 — **误报，已排除**
- **重新审计**: `ts_scan()` 对每个记录执行 `delta_time = time_base + time_delta` 解码。`time_base` 从扇区头部读取（每个扇区独立）。查询过滤使用解码后的绝对时间 `abs_time = hdr.time_base + rh.time_delta` 与 `ts_min/ts_max` 比较。
- **结论**: 每个扇区的 time_base 在 seal 时独立写入，跨扇区查询正确使用各自的 epoch。epoch 边界正确。**无需修复**。
