## 一、项目全景分析

### 1.1 核心特对象与设计目标

RocketDB 是针对**资源受限嵌入式系统**的双模 Flash 存储引擎：

| 维度 | KVDB | TSDB |
|------|------|------|
| **用途** | 键值存储（少量 key 频繁更新） | 时序存储（追加式，环形淘汰） |
| **典型工况** | 设备配置、参数存储（10~50 keys） | 传感器采样、告警日志 |
| **核心机制** | 四阶段 GC + Phase 4 磨损均衡 | 环形 rotation + Epoch 管理 |
| **保证** | 掉电一致性、GC 确定性 | 时间戳单调性、查询完整性 |

### 1.2 技术栈与制约

| 类别 | 约束 |
|------|------|
| **语言** | C99（禁用 bool、动态内存） |
| **硬件** | Cortex-M0/M3/M4/M7 + SPI NOR Flash |
| **分区** | 单分区 ≤1MB，扇区 4KB（可扩展） |
| **外部依赖** | CRC16、Hash16、Flash 操作（用户实现） |

## 三、关键设计决策总结

### 3.1 十三项已确认决策

| 决策 | 描述 | 理由 | 风险 |
|------|------|------|------|
| **D1** | iter 严格 lock（禁止修改） | 单任务环境最适合 | 无快照支持 |
| **D2** | gc_reserve 固定计算 | 标准化配置 | 部分场景可能浪费空间 |
| **D3** | 单扇区记录限制 | 典型工况不需跨扇区 | 大 value 需应用层分段 |
| **D4** | Epoch 自动重置 | 防护优先 | 可能频繁 rotation |
| **D5** | Query 简单缓冲（无分页） | v0.0.2 不需 | 超大查询需多次调用 |
| **D6** | 被动淘汰通知（rdb_tsdb_time_range） | 应用可主动查询 | 无主动淘汰告警 |
| **D7** | CRC 错误报告不隔离 | 故障可见性 | 坏数据可能继续被访问 |
| **D8** | CORRUPT 扇区跳过 | 暂时故障可恢复 | 永久故障需手动处理 |
| **D9** | yield 固定调用间隔 | 可预测性 | 实时性需要评估 |
| **D10** | CRC-16 而非 32 | 空间与检错平衡 | 检错率 99.99%「足够» |
| **D11** | write_seq 同时扫 create_seq | 防序列回退 | 初始化可能多扫一遍 |
| **D12** | 16 字节损坏跳过粒度 | 避免逐字节爬行 | 理论上可能跳过记录 |
| **D13** | time_delta uint32_t | rotation 由容量驱动 | 某些时序精度下可能预期 rotation |

### 3.2 抓住的设计要点

1. **防死锁兜底**：init Phase 4 强制恢复 gc_reserve + 1 个擦除扇区
2. **先写新再删旧**：ERR_FULL 时旧数据保持 VALID，不丢失
3. **虚拟垃圾计入**：will_free 参数提高 GC 预测准确性
4. **结构体自然对齐**：4 字节边界对齐，避免 Cortex-M0 HardFault
5. **损坏记录跳过策略**：16 字节步长，避免逐字节读取损坏区

## 八、最佳实践与推荐

### 8.1 应用集成建议

```c
/* 推荐的初始化流程 */
rdb_kvdb_t kvdb;
rdb_tsdb_t tsdb;
uint8_t kvdb_meta[128];
uint32_t tsdb_ec[8];

/* Phase 0：Init with recovery */
rdb_kvdb_init(&kvdb, &kvdb1_part, kvdb_meta);
rdb_tsdb_init(&tsdb, &tsdb1_part, tsdb_ec);

/* Phase 1：容量检查 */
uint32_t total, used, avail;
rdb_kvdb_space_info(&kvdb, &total, &used, &avail);
if (avail < critical_threshold)
    handle_low_space();

/* Phase 2：周期性 GC（可选）*/
if (rand() % 100 < gc_probability)
    rdb_kvdb_gc(&kvdb);  /* 手动触发，set 中会自动触发 */
```

### 8.2 性能优化建议

1. **避免频繁 format**：数据恢复 > 数据清零
2. **批量写入**：在 KVDB 中使用事务式更新（应用层实现）
3. **定期 reset_epoch**：TSDB 在跨 epoch 前主动重置
4. **监控统计信息**：定期检查 wear_info、stats，预测维护
