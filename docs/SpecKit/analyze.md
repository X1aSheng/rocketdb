## KVDB 功能覆盖

### 1. 生命周期管理

#### init()
- **设计**：扫描全部扇区、恢复状态、选择活跃扇区
- **阶段**
  - Phase 1：逐扇区 magic/hdr_crc 校验，状态分类（ERASED/ACTIVE/SEALED/CORRUPT）
  - Phase 2：依据 record seq 清理过期副本，重算 garbage_bytes
  - Phase 3：选择 create_seq 最大的活跃扇区
  - Phase 4：安全水位恢复（erased < gc_reserve + 1 时强制擦除）
- **覆盖**：✅ 完整（需验证 Phase 4 有效性）
- **风险**：Phase 4 依赖于 erase 成功；若扇区坏块可能卡住

#### format()
- **设计**：全盘擦除，erase_cnt 累加保留
- **实现**：✅ 完整
- **应用场景**：不频繁使用；主要用于生产烧录和故障恢复

### 2. 基本操作

#### set()
- **流程**：写新记录（WRITING 状态） → CRC 校验 → 提交（VALID） → 若空间触发自动 GC
- **覆盖**：✅ 完整
- **边界**：
  - ✅ 超大 value (MAX_VAL_LEN) 拒绝
  - ✅ 活跃扇区满时 rotate / GC
  - ⚠️ GC 失败时错误路径需验证
- **假设**：key 必须非空且 ≤ MAX_KEY_LEN；值可为 0 字节

#### get()
- **流程**：遍历全区查找最新有效副本 → CRC 校验 → 数据拷贝
- **覆盖**：✅ 完整
- **边界**：
  - ✅ 不存在返回 NOT_FOUND
  - ✅ 缓冲不足返回 TOO_LARGE，out_len 仍返回实际
  - ⚠️ CRC 失败与恢复流程
- **性能**：线性扫描，最坏 O(N)（N 为记录总量）

#### delete()
- **流程**：查找所有 VALID 副本 → 标记为 DEAD
- **覆盖**：✅ 完整
- **语义**：标记而非立即删除，下次 GC 时回收空间
- **幂等性**：✅ 重复 delete 相同 key 返回 OK

#### exists()
- **流程**：快速查找（跳过 CRC 校验）
- **覆盖**：✅ 完整
- **用途**：快速存在性检查，不需完整 CRC 校验

### 3. 垃圾回收

#### 四阶段评分 GC

**Phase 0：Zero-Live 扇区**
- 条件：garbage_bytes == sector 总容量（全部记录被删除）
- 操作：直接 erase，开销最小
- 覆盖：✅ 完整
- 风险：📊 需验证在连续 delete 工作负载下的触发频率

**Phase 1：Low-Live 扇区**
- 条件：垃圾率 > RDB_GC_GARBAGE_PCT（默认 20%）
- 操作：迁移有效记录到活跃扇区，然后 erase
- 覆盖：✅ 完整
- 复杂度：O(live_records)

**Phase 2：Wear 均衡**
- 条件：当 Phase 0/1 无新目标时，选择 erase_cnt 最小的扇区
- 操作：迁移 live_bytes，erase
- 覆盖：✅ 完整
- 目的：防止某些扇区频繁被擦除

**Phase 3：Oldest 扇区**
- 条件：所有扇区都在磨损均衡范围内
- 操作：选择 create_seq 最小的（最老）扇区，迁移、erase
- 覆盖：✅ 完整
- 影响：稳态 GC，开销相对较高

**Phase 4：冷扇区均衡（静态磨损均衡）**
- 条件：距上次 GC ≥ RDB_GC_WEAR_THRESHOLD 次的扇区
- 操作：主动 erase，无需迁移（Phase 0 优化）
- 覆盖：✅ 设计完整，⚠️ 实际效果未验证
- 关键：防止"长期不更新的冷扇区磨损远低于热扇区"的问题

**GC 触发**
- 自动触发：set() 检测空间不足时调用
- 手动触发：应用显式调用 rdb_kvdb_gc()
- 覆盖：✅ 完整

#### GC 前可行性校验
- **现状**：迁移前完整扫描判断目标可用空间
- **防死锁**：不会出现"中途无处可写"的永久死锁
- **覆盖**：✅ 设计层完整，需压力测试验证

### 4. 迭代与查询

#### iter_init/iter_next
- **设计**：遍历所有 VALID 记录；迭代期间禁止修改（代数检查）
- **覆盖**：✅ 完整
- **风险**：
  - ⚠️ 迭代期间修改返回 RDB_ERR_BUSY（严格模式）
  - ⚠️ 大数据集迭代性能未基准化

### 5. 统计与诊断

#### space_info / wear_info / stats
- **覆盖**：✅ 基础完整
- **用途**：应用层监控、磨损分析
- **缺口**：
  - 缺少 per-sector 的详细统计（某些高级调试需要）
  - 缺少实时 live_bytes 追踪的验证测试

## TSDB 功能覆盖

### 1. 生命周期管理

#### init()
- **设计**：扫描环形分区、定位 head/tail、恢复 time_base 和 epoch
- **阶段**
  - 扇区状态分类（ERASED/ACTIVE/SEALED）
  - 查找最新读和写指针
  - 恢复 epoch 和 time_base
- **覆盖**：✅ 完整
- **风险**：多个 ACTIVE 扇区时的选择逻辑

#### format()
- **设计**：全盘擦除，重置 head/tail/epoch
- **覆盖**：✅ 完整

### 2. 基本操作

#### append()
- **流程**：检查时间戳单调性 → 若回绕 epoch++ → 写新记录 → 若扇区满 rotation
- **覆盖**：✅ 完整
- **边界**：
  - ✅ 超大 data (MAX_TS_DATA_LEN) 拒绝
  - ✅ 扇区满自动 rotation 到下一扇区
  - ✅ 时间戳回绕自动 epoch++
- **约束**：时间戳必须 单调递增（回绕检测自动处理）

#### rdb_tsdb_query() / rdb_tsdb_query_ex()
- **流程**：遍历已封存的扇区 + 活跃扇区，查找 [from_ts, to_ts] 范围内的记录
- **覆盖**：✅ 完整
- **支持**：
  - ✅ 单扇区查询
  - ✅ 跨扇区查询
  - ⚠️ 跨 epoch 查询（需验证边界情况）
- **缺口**：
  - 缺分页/游标支持（大结果集需客户端循环调用）
  - 缺索引加速（全扇区线性扫描）

#### rdb_tsdb_time_range()
- **设计**：返回当前可查询的 [min_ts, max_ts]
- **用途**：应用层了解数据窗口，判断是否已过期
- **覆盖**：✅ 完整

### 3. Rotation 与 Epoch

#### Rotation（环形覆盖）
- **触发**：扇区满后自动切换到下一扇区
- **设计**：head/tail 指针环形推进
- **覆盖**：✅ 完整
- **风险**：
  - ⚠️ 多个循环后的 head/tail 一致性
  - ⚠️ 覆盖通知缺失（应用通过 rdb_tsdb_time_range 被动获知）

#### Epoch 管理
- **触发**：时间戳回绕（ts_n < ts_n-1）
- **语义**：新 epoch 下时间戳重新从 0 开始
- **查询**：跨 epoch 的数据应被找到
- **覆盖**：✅ 设计完整，⚠️ 跨 epoch 查询边界未完全验证
- **约束**：epoch 必须 <= 255（目前设计未具体限制，需明确）

### 4. 恢复与一致性

#### Recovery
- **设计**：init 时根据 magic/seq 恢复状态
- **覆盖**：✅ 基础完整
- **风险**：
  - ⚠️ 多个扇区 magic 都有效时的冲突解决
  - ⚠️ 部分写入（header/data 不同步）的恢复

## 架构设计评价

### 优点

1. **安全性优先**
   - 二次写（WRITING → VALID → DEAD）确保掉电一致性
   - Phase 4 安全水位防死锁
   - 显式错误码、无隐式隐瞒

2. **磨损均衡**
   - 四阶段 GC + Phase 4 动态均衡
   - erase_cnt 分布相对均匀（设计目标 ≤1000 差异）

3. **零动态内存**
   - API 面向嵌入式无堆资源约束
   - 所有缓冲由 caller 提供

4. **灵活性**
   - 支持多种 Flash 硬件（NOR、NAND 等）
   - yield 回调主动让出 CPU

### 缺点

1. **查询效率低**
   - KVDB get() 线性扫描，无索引
   - TSDB query 全扇区线性查询
   - 大数据集性能下降

2. **GC 复杂度**
   - Phase 3/4 选择逻辑复杂，维护成本高
   - 实际 Phase 4 效果未充分验证

3. **迭代严格**
   - iter 期间禁止修改（新增的 RDB_ERR_BUSY）
   - 某些应用场景可能需要快照隔离

4. **诊断不友好**
   - 缺少 per-sector 详细信息
   - 缺少 GC 决策的日志输出

## 性能特征分析

### 预期性能（基于设计）

| 操作 | 最好情况 | 最坏情况 | 平均情况 |
|------|---------|---------|---------|
| KVDB set | O(1) | O(N*M)* | O(log N) |
| KVDB get | O(1) | O(N) | O(log N) |
| KVDB delete | O(1) | O(N) | O(log N) |
| KVDB GC Phase 0~2 | O(k) | O(N) | O(k) |
| KVDB GC Phase 3/4 | 不触发 | O(N) | 低频 |
| TSDB append | O(1) | O(1) | O(1) |
| TSDB query | O(k) | O(N*M) | O(k) |

*N：总记录数，M：活跃扇区数，k：结果集大小

### 需基准化的指标

- [ ] 平均 set 延迟（无 GC）
- [ ] GC 阶段各自的延迟分布
- [ ] yield 调用频率与 CPU 让出时间
- [ ] 1000 key、平均 value 100B 的查询速度
- [ ] TSDB 10000+ 条数据的 range query 延迟

## 与参考实现的对比

### FlashDB (armink/FlashDB)
- FlashDB：KV 存储，采用链表 + 哈希索引加速
- RocketDB：采用日志结构，无索引（v0.0.2 限制）
- **劣势**：查询慢
- **优势**：GC 逻辑简洁、磨损均衡显式

### LittleFS (littlefs-project/littlefs)
- LittleFS：文件系统，NAND 友好
- RocketDB：KV/TS 存储，NOR 友好
- **对比**：应用域不同；RocketDB 的 Phase 4 类似 LittleFS 的磨损均衡思想

## 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Phase 4 不触发 | 低 | GC 不均衡，磨损不均 | 强制测试，调整阈值 |
| CRC 校验不足 | 低 | 漏报坏块 | 可选高阶校验，监控 stats |
| 掉电时脏数据 | 低 | 恢复失败 | 完整 recovery 测试 |
| 性能低于预期 | 中 | 实时性不满足 | 预基准化，优化关键路径 |
| 多分区冲突 | 低 | 数据混乱 | 严格分区隔离测试 |
