## 待确认问题

本节列出设计与实现中的开放性问题，需团队或架构师做出最终决策。

### KVDB 设计问题

#### 问题 1：iter_gen 代数锁定的严格程度

**描述**
- 当前实现：迭代期间若数据库被修改（set/delete），iter.gen != db.iter_gen，返回 RDB_ERR_BUSY。
- 问题：是否需要"幽影读"支持（迭代快照，允许并发修改）？

**选项**
1. 保持当前严格模式（迭代期间禁止修改）— 简单、确定性
2. 支持快照隔离（复杂度高，需支持多版本）— 提高并发性
3. 折中：warning 告警但允许继续迭代 — 可能导致数据不一致

**当前假设**
✅ 保持严格模式（iter 期间禁止修改）。原因：嵌入式系统通常单任务，不需复杂快照。

---

#### 问题 2：GC Reserve 扇区数的可配置性

**描述**
- 当前配置：`gc_reserve` 根据 sector_cnt 计算（>= 16 ? 2 : 1）。
- 问题：是否应暴露为编译时/运行时可配参数？

**选项**
1. 保持固定计算（当前） — 简化，开销最小
2. 编译时 `#define RDB_KV_GC_RESERVE` — 提高灵活性，编译多份
3. 运行时在 init 或 format 时传入 — 最灵活，但增加 API 复杂度

**当前假设**
✅ 保持固定计算。原因：大多数场景下一致的安全水位比较重要。如有特殊需求可阶段 2 时重新讨论。

---

#### 问题 3：超大 value 的跨扇区支持

**描述**
- 当前限制：单条记录 (hdr + key + value) 不能超过单扇区容量 (DEFAULT `< 4KB`)。
- 问题：是否支持跨扇区大记录？

**选项**
1. 保持单扇区限制（当前） — 简化实现、GC 清晰
2. 支持跨扇区大记录 — 提高容量，但 GC 复杂、垃圾率不好控
3. 分段存储大 value（key + chunk indices）— 折中方案

**当前假设**
✅ 保持单扇区限制。原因：典型 IoT 场景 value 较小（<256B）；大 value 场景可用分段存储替代。

### TSDB 设计问题

#### 问题 4：Epoch 回绕的自动 vs. 手动触发

**描述**
- 当前实现：时间戳检测到倒序（ts_n < ts_n-1）自动 epoch++。
- 问题：是否需要手动 epoch 重置（如 MCU 重启后需同步系统时间）？

**选项**
1. 保持自动（当前） — 对抗硬件时钟漂移
2. 提供 rdb_tsdb_set_epoch(db, new_epoch) — 手动控制，复杂度增加
3. 混合：自动检测倒序，同时提供手动 reset — 最安全，但文档需清楚

**当前假设**
✅ 保持自动，同时预留 rdb_tsdb_set_epoch() 接口（v1.0）。原因：自动防护更安全；手动重置在 recovery 时可能需要。

---

#### 问题 5：Query 结果缓冲大小限制

**描述**
- 当前实现：`rdb_tsdb_query_by_time()` 返回结果数受 caller 提供的 `result_buf` 大小限制。
- 问题：超大查询范围（>10000 条记录）如何处理？

**选项**
1. 要求 caller 预分配足够缓冲（当前） — 简单，caller 需自行管理
2. 分页查询接口（游标） — 支持流式读取，复杂度高
3. 链式回调（callback 遍历）— 内存高效，但 async 复杂

**当前假设**
✅ 保持当前简单模式。缓冲溢出时返回 RDB_ERR_TOO_LARGE，caller 可重复调用获取子集。分页支持在 v1.0 如有需要可补充。

---

#### 问题 6：环形覆盖的淘汰通知

**描述**
- 当前实现：TSDB rotation 时最旧扇区被覆盖，无显式通知。
- 问题：应用如何得知数据已被覆盖？

**选项**
1. 被动（当前）：应用自行维护保留周期，查询返回 NOT_FOUND 时即已过期
2. 主动通知：提供 rdb_tsdb_get_last_sealed_time() 告知最旧数据时间戳
3. 回调通知：rotation 前调用 user callback 告知即将删除的时间范围

**当前假设**
✅ 被动通知，结合 get_range() 接口使用。应用应定期调用 get_range() 获取当前可查范围；若需要的数据已不在范围内，则已被覆盖。

### 错误处理问题

#### 问题 7：CRC 错误的降级处理

**描述**
- 当前实现：数据 CRC 失败返回 RDB_ERR_CRC，记录 stats.crc_errors++，但不自动隔离或修复。
- 问题：是否应自动标记损坏记录为 DEAD，或上报给 GC？

**选项**
1. 保持当前：仅报告错误，应用决策（当前）— 最保险，但应用需处理
2. 自动隔离：CRC 失败自动标 DEAD（仅 get 失败，但 delete 时认为存在）
3. GC 清理：收集 CRC 失败的记录，下次 GC 时清除

**当前假设**
✅ 保持当前：仅报告错误。原因：CRC 常见于硬件问题（bit flip）或初始化不完整（掉电）；自动隐瞒可能掩盖真实问题。建议应用在 CRC 错误达到阈值时主动 format 或通知管理员。

---

#### 问题 8：sector 损坏的隔离策略

**描述**
- 当前实现：初始化时发现 CORRUPT 扇区，尝试 erase；若 erase 失败保留为 CORRUPT。
- 问题：运行时如何处理 CORRUPT 扇区（写入时是否跳过）？

**选项**
1. 跳过 CORRUPT（当前）：CORRUPT 扇区从可写候选池中排除，GC reserve 不计入
2. 强制清理：多次 erase 失败后标记为"坏块"，永久不使用
3. 降级（仅读）：允许读取但禁止写入

**当前假设**
✅ 跳过 CORRUPT（仅不用于新写）。理由：某些 Flash 芯片可能有暂时硬故障（ESD 干扰后恢复），跳过而非永久标记更安全。GC 时自动隔离。

### 性能与配置问题

#### 问题 9：yield 回调的调用频率

**描述**
- 当前实现：GC 和 rotation 期间定期（通常每 4KB 处理后）调用 yield()。
- 问题：是否需要可配置的 yield 调用间隔，以平衡实时性与性能？

**选项**
1. 保持固定间隔（当前）— 简单
2. 编译时配置 `RDB_YIELD_INTERVAL` — 提高灵活性
3. 运行时配置 — 最灵活，但增加 API 复杂度

**当前假设**
✅ 保持固定间隔。大多数场景下固定值可满足需求。

### 设计决策记录（来自 design.md）

#### 决策 10：write_seq 恢复策略

**问题**
- init Phase 1 如何恢复全局 write_seq？
- 风险：format 完成后无记录即掉电，若仅扫描 record_seq 则 write_seq=0，导致后续 create_seq 冲突

**解决方案**
同时扫描扇区头的 `create_seq` 和记录的 `seq`，取两者最大值作为 `write_seq` 初始值。

**理由**
防止序列号回退场景，确保任何掉电时刻都能正确恢复。

---

#### 决策 11：损坏记录跳过策略

**问题**
- scan_sector 遇到不合法 record header 时如何跳过？
- 逐字节爬行：穿越损坏区域需数百次无效 Flash 读取
- 虚假解析：步长过小可能对齐到记录中间的字节序列

**解决方案**
跳过 `ALIGN_UP(KV_REC_SZ, WR_GRAN)` 字节（最小 16 字节），即一个完整记录头宽度。

**理由**
1. 避免逐字节爬行，加速 init 和扫描
2. 降低虚假解析风险（16 字节步长使概率降至可忽略）

---

#### 决策 12：数据安全优先原则（"先写新再删旧"）

**设计要点**
1. `ensure_space` 通过 `will_free` 参数将旧记录空间计入 GC 的虚拟可用空间
2. GC 后仍无法腾出足够空间，直接返回 `ERR_FULL`，旧记录保持 VALID
3. 只有新记录成功提交后，才在最后作废旧副本
4. 任何掉电时刻，用户数据要么是旧值，要么是新值，永不丢失

**ERR_FULL 语义**
分区确实已满，调用者需要先删除部分 key 来释放空间。这是明确的、可处理的状态，优于静默丢数据。

---

#### 决策 13：TSDB time_delta 使用 uint32_t

**问题**
- 单扇区内时间跨度限制？
- uint16_t：最大 65535 秒（约 18 小时）
- uint32_t：最大 0xFFFFFFFE 秒（秒精度约 136 年，毫秒精度约 49.7 天）

**解决方案**
使用 `uint32_t` 作为 time_delta 类型。

**理由**
rotation 几乎仅由扇区空间写满驱动，从根本上消除因时间差过大引发的非预期 rotation。

### 混合长度测试要求

**问题**
是否需要混合不定长 KEY 和 data 的测试？

**答案**
✅ 是。测试必须包含随机混合的 key/value 长度，模拟真实场景。

**数据分布**（test_plan.md 4.2.1）：
- Key：50% 短(1-8B)，30% 中(9-32B)，15% 长(33-63B)，5% 最大
- Value：40% 小(0-32B)，40% 中(33-256B)，15% 大(257-1024B)，5% 最大

### GC 压力测试要求

**问题**
每个分区需要触发多少次 GC？

**答案**
✅ 至少 100 次。确保所有扇区都至少经历一次 GC 循环，验证磨损均衡。

### 写入粒度测试要求

**问题**
是否需要测试不同写入粒度？

**答案**
✅ 是。必须测试 write_gran = 0/1/2/3（对应 1/2/4/8 字节），确保对齐和填充逻辑正确。

**选项**
1. 固定间隔（当前） — 简单可预测
2. 自适应：根据 GC 进度自动调节 — 复杂，收益不大
3. 编译时宏配置 — 灵活，但编译多份

**当前假设**
✅ 保持固定间隔。建议固定为 4KB 处理量后调用一次。如需调整可编译时配置。

#### 问题 10：CRC16 vs. CRC32 权衡

**描述**
- 当前实现：使用 CRC-16-MODBUS（2 字节）。
- 问题：16 位 CRC 是否足够检测错误，或需升级到 CRC-32？

**选项**
1. 保持 CRC-16（当前） — On-Flash 空间紧张，16 位足以检测单扇区内大多数错误
2. 升级 CRC-32 — 检测水平更高，但 On-Flash 开销+2 字节/record
3. 混合：关键header用32，data用16 — 复杂度不值得

**当前假设**
✅ 保持 CRC-16。原因：CRC-16 对单扇区内的错误检测足够（≥99.99%）；On-Flash 空间宝贵；大多数实装系统运行环境相对稳定。重要应用可在应用层补充高阶检验（如 Hamming code）。

### 兼容性与版本问题

#### 问题 11：格式向前后兼容性

**描述**
- 当前实现：v0.0.2 格式与 v0.0.1 不兼容。新版本是否需要支持旧版本数据迁移？

**选项**
1. 不支持向后兼容（当前） — 简化设计，用户需预升级
2. 版本检查 + 自动升级 — 复杂，需迁移逻辑
3. 版本标签 + 提供迁移工具（单独程序） — 折中

**当前假设**
✅ 不支持（constitution 中明确说明）。v0.0.2 已经大幅改进 GC 与 recovery 逻辑，直接支持升级意义不大；需升级用户应自行 dump+重新初始化。
