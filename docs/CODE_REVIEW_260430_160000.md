## Executive Summary

RocketDB is a well-engineered, zero-allocation dual-mode Flash storage engine for embedded systems. The architecture is sound: log-structured KV store with four-phase scored GC and ring-buffer TSDB with epoch management. Power-loss consistency is a first-class design concern.

**Test Results:** All 38 test cases pass with 57,816+ assertions across 7 test suites. Zero compiler warnings with `-Wall -Wextra`.

**Key Finding:** The engine core is robust. Most defects are in edge cases (sector counts >127, large sectors >64KB), test quality (fault injection tests are nearly assertion-free), and documentation accuracy (outdated SpecKit, incorrect type references throughout guides).

## 1. Source Code Review

### 1.1 KVDB Engine (`src/rocketdb_kvdb.c`)

#### BUG-H1: `int8_t` overflow in `fixup_stale` when sector count > 127

**Severity:** High | **Lines:** ~902–904

```c
int8_t i = (int8_t)(cnt - 1);
for (; i >= 0; i--) {
    scan_sector(db, order[(uint8_t)i], fixup_cb, &fx, RDB_FALSE);
}
```

When `cnt` > 127, `(int8_t)(cnt - 1)` wraps negative, and the loop body never executes. This silently skips deduplication for sectors at indices 128+ when sector count exceeds 127. `RDB_MAX_SECTORS` is 255, so this is reachable.

**Impact:** Stale duplicate records in high-sector-count partitions survive dedup until the next GC pass, wasting space.

**Fix:** Change `int8_t` to `int16_t`.

---

#### BUG-H2: `uint16_t write_off` overflow for sectors > 64 KB

**Severity:** High | **Lines:** 577 (struct), 1045, 1256, 1508

The `write_off` field in `rdb_kv_sector_meta_t` is `uint16_t`, but sector sizes can be any power of 2 (e.g., 128 KB on W25Q128). For sectors > 64 KB, `write_off` silently truncates. Every place the field is set uses explicit `(uint16_t)` casts that suppress detection.

**Impact:** On 128 KB+ sectors, writes beyond 64 KB offset will wrap to near-zero offsets, corrupting the active sector.

**Fix:** Change `write_off` to `uint32_t` in `rdb_kv_sector_meta_t`.

---

#### BUG-H3: Missing `iter_gen` increment on zero-live GC erase

**Severity:** High | **Lines:** ~1388–1406

In `gc_execute`, the victim sector is erased at line 1388. `iter_gen` is incremented at line 1406 only when `gc.migrated > 0`. If the victim contains only DEAD records (migrated = 0), `iter_gen` is NOT bumped despite sector erasure.

**Impact:** Active iterators see the pre-GC generation and may access an erased (all-0xFF) sector, reading garbage data as if it were valid.

**Fix:** Increment `iter_gen` unconditionally after erasing a sector.

---

#### BUG-H4: Shared dedup set corrupts cross-sector dedup in `gc_execute`

**Severity:** High | **Lines:** ~1408–1418

`dedup_mark_cb` uses a shared `rdb_dedup_set_t` across ALL sectors during the post-GC dedup scan. The dedup set is not reset between sectors. A key first sighted in sector 5 (inserted into the hash set) will cause its copy in sector 3 (scanned later) to be marked DEAD — even if sector 3 has the newer copy.

**Impact:** If scanning order doesn't match sector age order, the wrong (newer) copy may be invalidated, causing data loss.

**Fix:** Sort sectors by `create_seq` before the dedup scan, matching `fixup_stale`'s approach.

---

#### BUG-M1: Unchecked `fl_read` return in `iter_next` value read

**Severity:** Medium | **Lines:** ~2815

```c
fl_read(db, base + cur_off + KV_REC_SZ + ka, val_buf, vr);
```

The return value is not checked. A flash read failure leaves uninitialized data in `val_buf`.

**Fix:** Check return value, return `RDB_ERR_FLASH` on failure.

---

#### BUG-L1: Sector header CRC excludes `erase_cnt` and `create_seq`

**Severity:** Low | **Line:** 385

```c
h.hdr_crc = rdb_crc16(&h, 6); /* CRC covers magic(4) + version(2) */
```

The CRC protects only 6 of 16 header bytes. The `erase_cnt` and `create_seq` fields (8 bytes) are unprotected. A bit-flip in these fields passes the CRC check and propagates into the runtime state.

**Impact:** Low probability, but a silent wear-accounting error if it triggers.

**Fix:** Extend CRC to cover all written fields (12 bytes: magic + version + erase_cnt + create_seq).

---

#### BUG-L2: False dedup positives for hash + prefix collisions

**Severity:** Low | **Lines:** ~807–809

For keys > 8 bytes, only the first 8 bytes are stored in the dedup prefix. Two different keys with the same hash, same length, and same first 8 bytes are incorrectly classified as "already seen."

**Impact:** Very low probability, but when it triggers, the older copy is marked DEAD.

---

#### BUG-L3: `is_erased` 3-probe heuristic insufficient

**Severity:** Low | **Lines:** 318–342

Only start, middle, and end of the sector are probed. A sector with a single non-0xFF byte at an unprobed location is misclassified as ERASED.

**Impact:** Low — the bad byte would be detected on first read attempt.

### 1.2 TSDB Engine (`src/rocketdb_tsdb.c`)

#### BUG-H5: `ts_rotate` failure path breaks ring state

**Severity:** High | **Lines:** ~597–642

When `ts_init_sec` fails at line 629, the function returns `RDB_ERR_FLASH` but:
- The old head sector was already sealed (line 599)
- `db->head_sec` was NOT updated (still points to sealed sector)
- `db->head_off` is stale

**Impact:** Subsequent appends write to a stale offset in a sealed sector, corrupting flash.

**Fix:** Restore `db->head_sec` and `db->head_off` before returning error, or defer the seal until after init_sec succeeds.

---

#### BUG-M2: Transient flash read errors classify sector as CORRUPT

**Severity:** Medium | **Lines:** 226, 236, 242

Any flash read failure during `ts_classify` permanently marks the sector as `TS_CORRUPT`. On init, corrupt sectors are erased — discarding potentially recoverable data.

**Impact:** A single bus glitch can destroy an entire sector's data at boot.

**Fix:** Retry reads (2–3 attempts) before declaring CORRUPT.

---

#### BUG-M3: `twr_f` return values ignored during WRITING recovery

**Severity:** Medium | **Lines:** 552, 557, 615

During `ts_scan`'s WRITING record recovery, the `twr_f` calls for state promotion/demotion never check return values. Flash write failures are silently ignored and `flash_errors` is not incremented.

**Fix:** Check return values and increment `stats.flash_errors`.

---

#### BUG-M4: `tdc()` unsigned wraparound risk

**Severity:** Medium | **Line:** 70

```c
return db->sector_size - tds(db);
```

If `tds(db)` > `sector_size` (corrupt partition params), this wraps to a very large `uint32_t`. Currently guarded by init validation, but fragile.

**Fix:** Add an assertion or early return for this invariant.

---

#### BUG-M5: Large stack allocation in `rdb_tsdb_format`

**Severity:** Medium | **Line:** 1039

```c
uint32_t saved_ec[RDB_MAX_SECTORS]; // 255 * 4 = 1020 bytes
```

1 KB stack allocation is dangerous on resource-constrained MCUs with limited stack (e.g., 2 KB total on some Cortex-M0).

**Fix:** Use smaller stack buffer with loop, or document stack requirement.

---

#### BUG-M6: No `tyield` during `ts_data_crc` loop

**Severity:** Medium | **Lines:** 176–189

For large payloads, the CRC loop reads in `RDB_STACK_BUF_SIZE` chunks without yielding. On slow SPI flash, this can starve an RTOS for hundreds of milliseconds.

**Fix:** Call `tyield(db)` after each chunk read.

---

#### BUG-L4: `get_latest`/`get_oldest` unchecked `trd` return for data copy

**Severity:** Low | **Lines:** 1726, 1843

The final data copy to caller buffer uses `trd()` without checking the return value. If the flash read fails after CRC verification passed, the caller gets garbage data.

**Fix:** Check return value and return `RDB_ERR_FLASH` on failure.

### Test Code Defects

#### BUG-T1: `test_fault_injection.c` missing `rdb_kvdb_format()` before `rdb_kvdb_init()`

**Severity:** High | **File:** `test/sim/test_fault_injection.c`, lines 77, 131, 185, 238, 297

All 5 test cases call `rdb_kvdb_init()` without first calling `rdb_kvdb_format()`. Every other test file does format-then-init correctly. This means these tests operate on uninitialized flash — the database state is undefined and test results are meaningless.

**Impact:** Fault injection tests provide no actual coverage of RocketDB fault tolerance. Their "PASS" status is misleading.

**Fix:** Add `rdb_kvdb_format()` before `rdb_kvdb_init()` in all 5 cases.

---

#### BUG-T2: `test_fault_injection.c` tests always return 0 (always pass)

**Severity:** High | **File:** `test/sim/test_fault_injection.c`

4 of 5 test functions return 0 unconditionally:

- `fault_write_fail_nth` (line 98): `return 0`
- `fault_write_fail_probability` (line 153): `return 0`
- `fault_erase_fail` (line 206): `return 0`
- `fault_data_corruption` (line 316): `return 0`

Only `fault_power_loss` has a single `TEST_ASSERT_RDB_OK` (line 258). The other tests print diagnostics but make zero assertions about correctness.

**Impact:** These tests would report PASS even if the entire fault injection mechanism were disabled or broken.

**Fix:** Add meaningful assertions verifying the expected error code from each fault scenario.

---

#### BUG-T3: `test_framework.c` double `va_start` undefined behavior

**Severity:** Medium | **File:** `test/sim/test_framework.c`, lines 50–66

```c
va_start(ap, fmt);   // first va_start
vprintf(fmt, ap);
va_end(ap);
// ...
va_start(ap, fmt);   // second va_start on same ap — UNDEFINED BEHAVIOR
vfprintf(log_file, fmt, ap);
va_end(ap);
```

Per C99 7.15.1.3, a `va_list` that has been `va_end`-ed cannot be reused with `va_start`. The second `vfprintf` receives garbage arguments on most ABIs.

**Impact:** Log file output is corrupted — argument values in log output are wrong.

**Fix:** Use `va_copy` or reorder to vfprintf first.

---

#### BUG-T4: `test_framework.c` `get_time_ms` uses `clock()` (CPU time, not wall)

**Severity:** Low | **File:** `test/sim/test_framework.c`, line 42

`clock()` measures CPU time, not wall-clock time. On Windows it is wall-clock time; on POSIX it is CPU time. This makes timing inconsistent across platforms and can wrap around.

**Fix:** Use `gettimeofday()` / `QueryPerformanceCounter()` or a cross-platform wall-clock API.

---

#### BUG-T5: Direct flash memory access in tests

**Severity:** Low | **Files:** `test_kvdb_basic.c` (lines 300, 309), `test_tsdb_stress.c` (lines 141, 182, 207, 214)

Several tests directly read/write `g_flash.mem[]` instead of going through `sim_flash_read`/`sim_flash_write`. This bypasses the flash abstraction, including write-granularity checks.

**Impact:** Tests may pass in simulation but the same operations would fail on real hardware with non-byte write granularity.

**Fix:** Route all flash access through `sim_flash_*` functions.
