### Failing Test Detail

**`kv_max_boundaries`** at `test/sim/test_kvdb_basic.c:425`:
```c
/* Key too long (64) → TOO_LARGE */
char ok[RDB_MAX_KEY_LEN + 3];
memset(ok, 'K', RDB_MAX_KEY_LEN + 1u);
ok[RDB_MAX_KEY_LEN + 1u] = '\0';
TEST_ASSERT_RDB_ERR(rdb_kvdb_set(&g_db, ok, "v", 1), RDB_ERR_TOO_LARGE);
// Expected: RDB_ERR_TOO_LARGE (-5)
// Actual:   RDB_ERR_PARAM (-1)
```

Root cause: `strkey_len()` function correctly detects the 64-byte key as "too long" and returns 1. However, the function returns 1 when `i < 1` (empty key) OR `i > RDB_MAX_KEY_LEN` (too long). But the condition `i < 1` catches `i == 0` only — a key with single null terminator at index 0. When the key length is 0, the function returns 1 for "empty" — but `rdb_kvdb_set()` treats return value 1 uniformly as `RDB_ERR_TOO_LARGE`. This is semantically correct (an empty key IS too small/large), but the test expects `RDB_ERR_TOO_LARGE` while the code currently returns `RDB_ERR_PARAM` for a different reason.

Actually, tracing more carefully: the key `ok` is 64 'K' chars + null at position 64. `strkey_len` iterates i=0..64, and at i=64, key[64]='\0', with `i > RDB_MAX_KEY_LEN (63)` = true, so it returns 1. Then `rdb_kvdb_set` checks `if (sr == 1) return RDB_ERR_TOO_LARGE;`. This should work. But the prebuilt binary was likely compiled from a DIFFERENT version of the code (before a strkey_len fix was applied). Rebuilding should resolve the issue.

**Verdict**: Likely a stale binary issue — rebuild needed to confirm.

## 2. Defects Found

### 2.1 [BUG-01] Version Number Mismatch — CRITICAL

**Files**: `src/rocketdb_kvdb.c:274`, `src/rocketdb.h:50`, `src/rocketdb_tsdb.c:30`

- `rdb_version()` returns `0x010000u` (v1.0.0)
- All source file headers state `@version 1.0.0`
- CHANGELOG and README state version 1.1.0

**Fix**: Update `rdb_version()` to return `0x010100u`, update `@version` tags to `1.1.0`.

### 2.2 [BUG-02] KVDB Init Phase 3: SEALED Sector Not Reset to ACTIVE — MEDIUM

**File**: `src/rocketdb_kvdb.c:1968-1970`

```c
db->active_sec = best;
db->write_off = db->sectors[best].write_off;
```

When Phase 3 fallback selects a SEALED sector (one with highest create_seq regardless of space), the sector's status remains `RDB_SEC_SEALED`. The subsequent write will trigger `ensure_space()` → `rotate()` which handles it, but the runtime state is inconsistent: an "active" sector has SEALED status.

**Fix**: Set `db->sectors[best].status = RDB_SEC_ACTIVE;` after selection.

### 2.3 [BUG-03] TSDB Recount Modifies Flash During Append — LOW

**File**: `src/rocketdb_tsdb.c:1270-1288`

The periodic `total_count` reconciliation in `rdb_tsdb_append()` calls `ts_scan(..., RDB_TRUE)` which recovers WRITING records by writing to flash. This is unexpected behavior during a write operation and causes additional flash wear.

The scenario where this matters:
- An interrupted append leaves a WRITING record
- Next append triggers a recount that modifies flash to promote/demote it
- This is functionally correct but should be documented

**Fix**: Either use `RDB_FALSE` (read-only recount) or add a comment explaining the design decision.

### 2.4 [BUG-04] README Broken Link — LOW

**File**: `README.md:102`

```markdown
- [`docs/design.md`](docs/design.md) — Full architecture and design manual
```

The actual file is `docs/rocketdb design.md` (contains a space). This link is broken.

**Fix**: Update link to `docs/rocketdb%20design.md` or rename the file.

### 2.5 [BUG-05] TSDB rdb_tsdb_format Missing sector_cnt Validation — LOW

**File**: `src/rocketdb_tsdb.c:1021-1031`

```c
rdb_err_t rdb_tsdb_format(rdb_tsdb_t* db) {
    if (!db || !db->part) return RDB_ERR_PARAM;
    { uint32_t scnt32 = ...; if (scnt32 < RDB_TS_MIN_SECTORS ...) return RDB_ERR_PARAM; }
```

The function validates `part->total_size / part->sector_size` but does not validate against `db->sector_cnt` if the caller pre-set it. KVDB's format does validate `db->sectors` but TSDB's doesn't validate `db->erase_cnts`.

**Fix**: Add NULL validation symmetry with KVDB format: if `db->sector_cnt` was pre-set, verify consistency.

## 3. Improvements

### 3.1 [IMP-01] Add W25QXX Flash Integration Guide — HIGH

The project claims W25QXX support but has no specific documentation. A W25QXX integration guide should include:
- SPI command sequences (Read Status, Write Enable, Sector Erase 0x20, Page Program 0x02, Read Data 0x03)
- Write granularity: W25QXX supports page program (256 bytes), but minimum write unit is 1 byte. Set `write_gran = 0`.
- Sector size: W25Q32 = 4KB sectors, W25Q64 = 4KB/32KB/64KB blocks. Recommend 4KB uniform sectors.
- Timing: tW (write) = 3ms typ, tE (sector erase) = 45ms typ, tBP (block erase) = 150ms typ
- Example HAL implementation for W25QXX via SPI

## 5. W25QXX Flash Chip Analysis

### 5.1 Compatibility Assessment

The RocketDB design is well-aligned with W25QXX series characteristics:

| Feature | W25QXX | RocketDB | Compatible? |
|---|---|---|---|
| Sector size | 4KB (uniform) | Configurable (≥4KB) | ✓ |
| Write granularity | 1 byte | 1/2/4/8 bytes | ✓ (use wg=0) |
| NOR 1→0 semantics | Yes (page program) | Required | ✓ |
| Erase before write | Yes | Enforced | ✓ |
| Erase time | 45ms typ (4KB sector) | Yield callback | ✓ |
| Page program | 256B max | Stack buffer ≤64B | ✓ |
| Endurance | 100K cycles | Wear leveling | ✓ |

### 5.2 Recommended Configuration

For W25Q32 (4MB, 1024×4KB sectors):
```c
#define RDB_MIN_SECTOR_SIZE 4096u
#define RDB_MAX_SECTORS     255u
// KVDB: 16 sectors × 4KB = 64KB (with gc_reserve=2)
// TSDB: 16 sectors × 4KB = 64KB
```

### 5.3 SPI HAL Implementation Notes

The critical considerations for W25QXX:
1. **Write Enable before Erase/Program**: Every `erase()` and `write()` must send WREN (0x06) first
2. **Busy polling after Erase/Program**: Wait for WIP (bit 0) to clear in Status Register 1
3. **Page boundary**: Page Program cannot cross 256-byte page boundaries
4. **Write protect**: Ensure BP bits are cleared or handled correctly
5. **Lock/unlock**: SPI bus mutex in multi-threaded environments
