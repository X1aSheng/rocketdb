## Defect List

### P1 — High Severity

#### BUG-P1-01: `fixup_stale` sort produces wrong ordering for dedup pass

**File:** `src/rocketdb_kvdb.c:885-894`

The insertion sort in `fixup_stale` sorts sectors by `create_seq` in **ascending** order (oldest first), producing `order[0] = oldest, order[cnt-1] = newest`. But the dedup scan runs from `cnt-1` down to `0` — which is **newest→oldest** (correct).

However, the sort loop condition is:
```c
while (j > 0 && RDB_SEQ_GT(db->sectors[order[j - 1]].create_seq,
                             db->sectors[key].create_seq))
```
This moves larger `create_seq` values right → ascending (oldest first). Correct.

But the `dedup_mark_cb` sort in `gc_execute` (line 1423-1432) uses the INVERSE comparison:
```c
while (k >= 0 && RDB_SEQ_GT(db->sectors[key].create_seq,
                            db->sectors[order[k]].create_seq))
```
This moves larger values left → descending (newest first). Also correct for its purpose.

**Verdict:** Both sort orders are internally correct for their respective use cases. No bug — just confusingly inverted comparison patterns. Adding a comment would help.

**Resolution:** Add clarifying comment explaining the two sort orders. **(Documentation-only, no code change required.)**

### P2 — Medium Severity

#### BUG-P2-01: Makefile uses Windows cmd.exe syntax — not portable to Linux/macOS

**File:** `Makefile:158-161`, lines throughout

The Makefile uses:
- `@if exist` (line 57) — cmd.exe, not sh
- `@for %%t` (line 158) — cmd.exe batch syntax
- `rmdir /Q /S` (line 167) — cmd.exe

This prevents building on Linux/macOS where `make` typically uses bash.

**Fix:** Replace with POSIX-compatible shell commands.

---

#### BUG-P2-02: `sim_dist.c` uses `<math.h>` but Makefile lacks `-lm`

**File:** `Makefile` (all link rules), `test/sim/sim_dist.c:2`

`sim_dist.c` calls `sqrt()`, `log()`, `cos()`, `pow()`, `llround()` from `<math.h>`. On Linux/macOS, linking requires `-lm`. The Makefile link rules for `test_integration.exe` and `test_kvdb_basic.exe` (which include `sim_dist.o`) omit `-lm`.

**Fix:** Add `-lm` to link rules that include `sim_dist.o`.

---

#### BUG-P2-03: `rdb_kv_iter_next` truncates `sector_size` to `uint16_t`

**File:** `src/rocketdb_kvdb.c:2776, 2782`

When the iterator encounters a sector boundary or end-of-chain, it sets:
```c
it->offset = (uint16_t)ss;  // ss = sector_size (uint32_t)
```
For sectors > 64 KB (e.g., 128 KB), `(uint16_t)131072 = 0`, causing the loop condition `(uint32_t)it->offset + KV_REC_SZ <= ss` to always be true — the iterator loops infinitely.

**Fix:** Remove the `(uint16_t)` cast; `it->offset` is already `uint32_t`.

---

#### BUG-P2-04: `tdc()` guard insufficient for extreme sector sizes

**File:** `src/rocketdb_tsdb.c:733`

The guard `if (db->sector_size <= tds(db))` correctly catches sectors too small to hold a header. But `tds(db) = RDB_ALIGN_UP(TS_HDR_SZ, twr(db))`. With `write_gran=3` (8B), `TS_HDR_SZ=20`, `tds=24`. A 24-byte sector would be rejected. However, the minimum sector size is 4096 by default, so this is unreachable in practice. No action needed — just noting the guard works correctly.

### P3 — Low Severity

#### BUG-P3-01: TSDB `ts_classify` uses 2-point erase check vs KVDB's 3-point

**File:** `src/rocketdb_tsdb.c:242-257`

TSDB's erase verification probes only the **midpoint** and **last 4 bytes**. KVDB's `is_erased()` probes **start**, **midpoint**, and **last 4 bytes** (3-point check).

While TSDB also checks that `h.magic == 0xFFFFFFFFu` (4 bytes at offset 0), this is the same as KVDB's "probe 1" via a different mechanism. So functionally, TSDB does verify start+mid+end — but the start check is implicit via the magic field read, not an explicit erase probe. The code is functionally equivalent but the asymmetry is confusing.

**Fix:** Add a comment noting that the magic == 0xFFFFFFFF check serves as the start-of-sector erase probe.

#### BUG-P3-02: `test_make_log_path` uses static buffer — not thread-safe

**File:** `test/sim/test_framework.c:326-333`

Returns a pointer to a `static char path[128]`. Successive calls overwrite the buffer. This is a non-issue for single-threaded test runners but violates good practice.

**Fix:** Document the limitation or use caller-provided buffer.

#### BUG-P3-04: `rdb_dedup_slot_t.prefix[8]` stores only first 8 key bytes

**File:** `src/rocketdb_kvdb.c:753-757`

The dedup hash set uses a `prefix[8]` field for fast collision disambiguation. For keys sharing the same first 8 bytes, hash, and length, the table produces a false match, causing fallback to `find_latest()`. For keys > 8 bytes with common prefixes (e.g., `"sensor_001_value"` vs `"sensor_001_state"`), this degrades dedup performance.

**Verdict:** Not a correctness bug — fallback to `find_latest()` guarantees correctness. Performance impact is negligible for typical embedded key sizes (≤63 bytes). Acceptable tradeoff.

---

#### BUG-P3-05: `test_framework.c:61` — `va_copy` undefined behavior if `ap` is reused

**File:** `test/sim/test_framework.c:61`

```c
va_copy(ap_copy, ap);
vfprintf(g_config.log_file, fmt, ap_copy);
va_end(ap_copy);
```

After `va_copy(ap_copy, ap)`, `ap` has already been consumed by `vprintf` on line 57. Per C99 §7.15, the state of `ap` after `vprintf` is indeterminate. Using it as source for `va_copy` is technically undefined behavior. In practice, most implementations handle this fine, but it's technically incorrect.

**Fix:** Call `va_copy` BEFORE `vprintf`, or use `va_start` twice.
