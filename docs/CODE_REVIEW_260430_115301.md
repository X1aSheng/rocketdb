## 2. Remaining Defects

### 2.1 High Severity

#### BUG-H2: `uint16_t write_off` overflow for sectors > 64 KB

**Severity:** High | **File:** `src/rocketdb.h:577`, `src/rocketdb_kvdb.c`

The `write_off` field in `rdb_kv_sector_meta_t` is `uint16_t`, but sector sizes can be any power of 2. For sectors > 64 KB (e.g., 128 KB on W25Q128), write_off silently truncates at 65535. Every setter uses explicit `(uint16_t)` casts that suppress compiler warnings.

Affected locations:
- `rocketdb.h:577` — struct definition
- `kvdb.c:1045,1256,2335` — explicit (uint16_t) casts on assignment
- All comparisons: `write_off + rsz > sector_size` — works because rsz is uint32_t and write_off promotes

**Impact:** On 128 KB+ sectors, writes beyond 64 KB offset wrap to near-zero, corrupting the active sector. Since `RDB_MAX_SECTORS = 255` and sector_size must be `>= RDB_MIN_SECTOR_SIZE = 4096`, this is reachable on mid-to-large NOR flash parts (W25Q64 with 64KB blocks, W25Q128+).

**Fix:** Change `write_off` to `uint32_t` in `rdb_kv_sector_meta_t` and remove all `(uint16_t)` casts on assignment. Also change in `rdb_kvdb_t` at line 620 (currently `uint16_t write_off` with `uint16_t _pad`).

### 2.2 Medium Severity

#### BUG-M3a: Unchecked `twr_f` return in init Phase 3 WRITING recovery

**Severity:** Medium | **File:** `src/rocketdb_tsdb.c:917,927`

In `rdb_tsdb_init` Phase 3 (lines 910-928), the WRITING record recovery path calls `twr_f()` to promote/demote records but does NOT check return values:
```c
// line 917
twr_f(db, base + off + 1, &v, 1);  // return not checked
// line 927
twr_f(db, base + off + 1, &d, 1);  // return not checked
```
This is inconsistent with `ts_scan()` (lines 562, 568) which properly checks return values. A flash write failure here silently leaves the record in WRITING state, causing repeated recovery attempts on every init until the flash cell degrades beyond repair.

**Fix:** Check return values as done in `ts_scan()`:
```c
if (twr_f(db, base + off + 1, &v, 1) == 0)
    db->head_count++;
```

#### BUG-M4: `tdc()` potential unsigned wraparound

**Severity:** Medium | **File:** `src/rocketdb_tsdb.c:70`

```c
return db->sector_size - tds(db);
```
If `tds(db) > sector_size` (impossible under current init validation, but no compile-time guard), this wraps to a very large uint32_t. The sector_size validation in init checks `sector_size >= RDB_MIN_SECTOR_SIZE` but not `sector_size > TS_HDR_SZ + granularity padding`.

**Fix:** Add a `_Static_assert` or runtime assert in init.

#### BUG-M5: 1 KB stack allocation in `rdb_tsdb_format`

**Severity:** Medium | **File:** `src/rocketdb_tsdb.c:1056`

```c
uint32_t saved_ec[RDB_MAX_SECTORS]; // 255 × 4 = 1020 bytes
```
and an identical array in `rdb_kvdb_format` at kvdb.c:2072. On Cortex-M0 with 2 KB total stack, this consumes half the stack in a single frame. A stack overflow here causes silent memory corruption.

**Fix:** Either:
- Use `static` allocation with a mutex guard
- Reduce array size and process in batches
- Document the stack requirement prominently

#### BUG-T2: Fault injection tests return 0 unconditionally

**Severity:** Medium | **File:** `test/sim/test_fault_injection.c`

4 of 5 test functions have minimal or zero correctness assertions:

| Test | Assertions | Issue |
|------|-----------|-------|
| `fault_write_fail_nth` | `TEST_ASSERT(fail_count > 0)` | Only checks that something failed, not that recovery is clean |
| `fault_write_fail_probability` | Returns 0 | No assertions at all (0 assertions in report) |
| `fault_erase_fail` | Returns 0 | No assertions about GC behavior after erase failure |
| `fault_data_corruption` | Returns 0 | No assertions about CRC error detection |

Only `fault_power_loss` has `TEST_ASSERT_RDB_OK` verifying successful post-power-loss recovery.

**Impact:** These tests report PASS even if the fault injection mechanism itself is broken.

**Fix:** After each fault scenario, verify database consistency:
- For write failure: verify DB is still init-able and existing data is readable
- For erase failure: verify GC either succeeded or left data intact
- For data corruption: verify CRC error is returned on read

#### BUG-T5: Direct flash memory access in tests

**Severity:** Medium | **Files:** `test_kvdb_basic.c`, `test_tsdb_stress.c`

Tests directly access `g_flash.mem[]` instead of using `sim_flash_read`/`sim_flash_write`:
- `test_kvdb_basic.c:300` — `memcpy(&rh, g_flash.mem + base + off, ...)`
- `test_kvdb_basic.c:309` — `g_flash.mem[first_addr] = 0x00`
- `test_tsdb_stress.c:141,182,208,214` — direct mem reads/writes

This bypasses write-granularity enforcement and fault injection. On hardware, these byte-level operations may fail if write_gran > 0.

**Fix:** Use `sim_flash_read()` and `sim_flash_write()` for all flash access in tests.

### 2.3 Low Severity

#### BUG-L1: Sector header CRC excludes erase_cnt and create_seq

**Severity:** Low | **File:** `src/rocketdb_kvdb.c:385`

```c
h.hdr_crc = rdb_crc16(&h, 6); /* CRC covers magic(4) + version(2) */
```
Only 6 of 16 bytes are CRC-protected. A bit-flip in `erase_cnt` or `create_seq` passes CRC validation and silently corrupts wear accounting.

**Fix:** Extend CRC to cover all header fields (12 bytes: magic + version + erase_cnt + create_seq).

#### BUG-L2: False dedup positives for hash + 8-byte prefix collisions

**Severity:** Low | **File:** `src/rocketdb_kvdb.c:799-808`

For keys > 8 bytes, only the first 8 bytes are stored in the dedup hash set prefix. Two different keys with same hash, same length, and same first 8 bytes are classified as "already seen", marking the older copy DEAD.

**Probability:** Extremely low for normal keys. Higher for structured keys with common prefixes (e.g., file paths, sensor IDs with common roots).

**Fix:** Increase prefix storage or store full key hash (use 32-bit secondary hash).

#### BUG-L3: `is_erased` 3-probe heuristic

**Severity:** Low | **File:** `src/rocketdb_kvdb.c:318-342`

Only start, middle, and end of sector are probed. A single non-0xFF byte at an unprobed offset is missed.

**Impact:** Very low — the bad byte is detected on first read attempt of the supposedly-erased sector.

#### BUG-L4: Unchecked `trd` return in get_latest/get_oldest data copy

**Severity:** Low | **File:** `src/rocketdb_tsdb.c:1743,1860`

```c
trd(db, da, buf, rd);  // return value ignored
```
If flash read fails after CRC passed, the caller receives uninitialized/garbage data in `buf`.

**Fix:** Check return value and return RDB_ERR_FLASH on failure.

#### BUG-L5: `ts_find_last_valid` signature mismatch

**Severity:** Low | **File:** `src/rocketdb_tsdb.c:1626`

```c
static int ts_find_last_valid(const rdb_tsdb_t* db, ...
```
The `db` parameter is declared `const` but the function calls `trd()` which takes non-const `rdb_tsdb_t*`. This works only because of implicit const-discard — adding `-Wcast-qual` would flag it.

**Fix:** Match const-qualification or remove const.

#### BUG-L6: `test_framework.c` uses `clock()` (CPU time)

**Severity:** Low | **File:** `test/sim/test_framework.c:44`

`clock()` measures CPU time, not wall-clock. On POSIX systems this measures CPU time consumed (not real elapsed time). On Windows it approximates wall time. Cross-platform inconsistency.

**Fix:** Use `time()` or platform-specific wall-clock API.
