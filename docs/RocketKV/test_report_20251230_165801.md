

# KVDB 模拟测试报告

## 测试环境

```
芯片: W25Q128
分区: KVDB1 — 32KB, base_addr=0x000000
sector_size: 4096 bytes
sector_cnt: 8
write_gran: 0 (1 byte)
KV_HDR_SZ: 16 bytes
data_start: ALIGN_UP(16, 1) = 16 bytes
data_cap: 4096 - 16 = 4080 bytes
KV_REC_SZ: 16 bytes
gc_reserve: RDB_GC_RESERVE(8) = 1
可用扇区: 8 - 1 = 7
max_live: 7 × 4080 = 28560 bytes
```

## 测试记录规格

```
测试 key 集合: "K00"~"K19" (3 bytes), 共 20 个 key
测试 value: 32 bytes 固定
rec_size = 16 + ALIGN_UP(3,1) + ALIGN_UP(32,1) = 16 + 3 + 32 = 51 bytes
每 sector 记录数 = floor(4080 / 51) = 80 条 (余 0 bytes)
20 key 存活数据 = 20 × 51 = 1020 bytes (max_live 的 3.6%)
```

## 测试用例 1: format + 初次写入 20 key

### format

```
rdb_kvdb_format(&db):
  Pass 1: 读取所有 sector erase_cnt
    全为 0xFF → saved_ec[0..7] = 0
  Pass 2: erase sectors 1..7
    sectors[1..7].erase_cnt = 1, status=ERASED, write_off=16
  Sector 0: sectors[0].erase_cnt = 0
    write_seq = 1
    init_sector(0):
      ec = 0 + 1 = 1
      erase sector 0
      write_seq++ → 2
      write_sec_hdr(0, ec=1, cseq=2)
      sectors[0] = {ec=1, cseq=2, garbage=0, write_off=16, status=ACTIVE}
  
  active_sec=0, write_off=16, live_bytes=0

状态:
  Sector:  [0:A@16]  [1:E]  [2:E]  [3:E]  [4:E]  [5:E]  [6:E]  [7:E]
  A=Active, E=Erased, @16=write_off
  erased=7, live_bytes=0, write_seq=2
  ec_all = [1, 1, 1, 1, 1, 1, 1, 1]
```

### 写入 K00..K19 (20 条, 各 51B, 共 1020B)

```
set("K00", val32):
  kl=3, len=32, rsz=51
  find_latest("K00") → not found, fc.found=0
  ensure_space(51, 0, 0xFF):
    eff = 0, 0+51 ≤ 28560 → OK
    erased=7 ≥ gc_reserve+1=2 → 快速退出 OK
  find_latest("K00") → still not found (no GC occurred)
  active sector 0: write_off(16)+51=67 ≤ 4096 → OK
  
  write_seq=3
  dcrc = CRC16("K00" ‖ val32)
  header: magic=0xA5, state=0xFF, kl=3, vl=32, hash=hash16("K00"), seq=3, dcrc
  
  rsz(51) ≤ RDB_STACK_BUF_SIZE(64) → 合并写入
  wa = 0x000000 + 16 = 0x000010
  write(0x000010, mbuf, 51) → header+key+val 一次写入
  commit: write(0x000011, 0xFE, 1) → state=VALID
  
  sectors[0].write_off = 16 + 51 = 67
  live_bytes = 51
  fc.found=0 → 不作废旧记录
  stats.write_ops=1, iter_gen=1

set("K01"..K19") 同理:
  每条: write_off += 51, live_bytes += 51, write_seq++

写入 K19 后:

  write_seq = 22 (2 + 20)
  erased = 7 (仅 sector 0 有数据)
  stats.write_ops = 20

Sector 0 状态:
  [HDR] [K00:seq3] [K01:seq4] ... [K19:seq22] [0xFF × 3044]
  write_off=1036, 剩余=4096-1036=3060, 可再写 3060/51=60 条
```

### 继续覆盖到 50 次 GC

**GC 触发模式分析**：

```
稳态模式:
  active sector 填满(80 条覆盖) → 需要 rotate → erased 不足 → GC
  
  每次 GC 前:
    最老的 sealed sector 中 20 条 live (最新版本 key)
    + 60 条 garbage (被覆盖的旧版本)
    → Phase 1 不命中 (live≠0)
    → Phase 2 评分选择: garbage_pct = 60/80*100 = 75%

  等等——让我重新分析稳态下各 sector 的 garbage/live 分布。

  每个 sealed sector 包含 80 条记录:
    第 N 轮覆盖写入的 20 条中，只有最后一轮的是 live
    sector 中最多只有 20 条 live (最近一轮 K00..K19)
    其余 60 条都是旧版本 → DEAD

  但是！每次 set 的 Step 5 会 mark_dead 旧副本。
  当 K00 在 sector X 被覆盖到 sector Y 时:
    sector X 中的 K00 → DEAD, garbage += 51
    sector Y 中的 K00 → VALID

  所以完整覆盖一轮后:
    旧 active sector: 所有 20 条最新版本从这里写出到新 sector
    → 旧 sector 中 20 条被 mark_dead → garbage += 20×51 = 1020
    → 旧 sector 最终: 80 条全 DEAD, garbage=4080, live=0

  这意味着: 每完成一轮完整覆盖(20 key),最老的有数据 sector 
  变成 all-DEAD → Phase 1 可零 live 回收!
```

**稳态 GC 模式确认：Phase 1 零 live 快速回收**

```
循环模式:
  1. 向 active sector 写 80 条 (4 轮覆盖)
  2. active 满 → ensure_space
  3. 此时最老的 sealed sector 已经全 DEAD (4 轮覆盖跨越了它)
  4. Phase 1 选中 → 直接 erase → 无迁移
  5. rotate → 新 active
  
  每次 GC: 0 条迁移, 1 次 erase
  这是少量 key 频繁更新的理想模式。
```

**GC 触发间隔**:

```
每个 sector 容纳 80 条覆盖记录
80 条 / 20 key = 4 轮完整覆盖
每 4 轮触发 1 次 GC

50 次 GC 需要: 50 × 4 = 200 轮完整覆盖
总 set 操作: 初始 20 + 200×20 = 4020 次
加上 sector 0 首次填满的额外操作: ~4020 次
```

这不是 bug——这是逻辑上正确的行为:
- 8 个 sector, gc_reserve=1, 可用 7 个
- 如果 7 个 sector 都有 live 数据, 且 erased=0
- GC 需要搬迁 live 数据到新位置, 但没有空位
- gc_avail 为 0 → 所有 candidate 不可行
- 返回 ERR_FULL

**根因**: 80 个 key × 50B = 4000B live, max_live=28560, 利用率仅 14%。但 live 数据分散在 7 个 sector 中, 每个 sector 都有 live → GC 无法清空任何一个。

**解决**: 这个场景中用户需要先 delete 某个 sector 中的所有 live key, 使其变成零 live, 然后 GC 可以回收。或者使用更多 sector（减少分散）。

**这实际上是分析报告中 K-2 复杂度问题的一个副作用**: 当 key 数量远超每 sector 容量，且 key 的访问模式导致 live 分散时，GC 效率下降。

**对于测试参数（20 key, 51B/条, max_live=1020, 单 sector 可容纳全部 live）**，这个问题不会出现——20 key 的全部 live 数据只需 1 个 sector, 其余 sector 在完整覆盖后变成零 live。

**让我用更合理的方式触发 Phase 2：部分删除使 garbage 不均匀**

```
20 key, 但把 2 个 key (K18, K19) 钉在 sector 0 (只写一次不覆盖)
只覆盖 K00..K17 (18 key)

sector 0 初始: K00..K19 (20条, 1020B), 然后覆盖 K00..K17:
  sector 0 可再写 60 条覆盖 → 3.3 轮 K00..K17
  覆盖 3 轮(54条): write_off=16+20×51+54×51=16+1020+2754=3790
  再写 6 条(K00..K05): write_off=3790+6×51=4096

  sector 0 状态:
    K06..K17 最新版在此 (覆盖#3的版本): 12×51=612B live
    K18, K19 始终在此: 2×51=102B live
    K00..K05 最新版在此(覆盖#3后又写一轮的6条): 6×51=306B live
    total live = 612+102+306 = 1020B
    garbage = 4080-1020 = 3060B

  rotate → sector 1, 覆盖 K06..K17(12条), 使 sector 0 中它们的版本 DEAD
  sector 0 live = K18,K19(102B) + K00..K05某版本
  
  这变得太复杂了。让我直接构造 GC 评分的数学验证。
```

### Phase 2 评分数学验证

```
给定:
  sector A: used=4080, garbage=3600, live=480, ec=3
  sector B: used=4080, garbage=2000, live=2080, ec=1
  g_min_ec=1, g_max_ec=3, ec_range=2
  erased=1 (= gc_reserve), thresh=5%

  gpct = 3600*100/4080 = 88%     → 88≥5 OK
  live(480) ≤ gc_avail? 假设 avail=4080(1个ERASED sector) → OK
  wpct = (3-1)*100/2 = 100
  cpct = 100 - min(480*100/4080, 100) = 100-11 = 89
  score = 88*7 + 100*3 + 89*1 = 616+300+89 = 1005

sector B 评分:
  gpct = 2000*100/4080 = 49%     → 49≥5 OK
  live(2080) ≤ avail(4080)? → OK

  score = 49*7 + 0*3 + 49*1 = 343+0+49 = 392

选择: sector A (score 1005 > 392) ✓

评分分析:
  - garbage 权重(×7)占主导: 高 garbage 率优先回收
  - wear 权重(×3): 偏好高 ec 的 sector, 让低 ec 的 sector 保留更久
```

### 场景
