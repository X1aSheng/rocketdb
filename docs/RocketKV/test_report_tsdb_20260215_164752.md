

# TSDB 模拟测试报告

## 测试环境

```
芯片: W25Q128
分区: TSDB1 — 32KB, base_addr=0x010000
sector_size: 4096 bytes
sector_cnt: 8
write_gran: 0 (1 byte)
TS_HDR_SZ: 20 bytes
data_start (tds): 20 bytes (ALIGN_UP(20,1) = 20)
data_cap (tdc): 4096 - 20 = 4076 bytes
TS_REC_SZ: 12 bytes
```

## 测试记录规格

```
data_len = 16 bytes (模拟传感器采样)
rec_size = trs(db, 16) = 12 + ALIGN_UP(16, 1) = 28 bytes
每 sector 记录数 = floor(4076 / 28) = 145 条 (余 16 bytes < 28)
max_data_len = 4076 - 12 = 4064 bytes
```

## 测试用例 5: erase_cnt 单调性 (ts_init_sec)

### 场景

同一次运行中对 sector 5 执行两次 ts_init_sec（可能由 format 后立即触发 rotation 引起）。

```
初始: ec_buf[5] = 10

第一次 ts_init_sec(5, seq=20):
  读 flash header → magic=TSDB, erase_cnt=10
  old_ec = max(ec_buf[5]=10, flash=10) = 10
  erase → write header(ec=11)
  ec_buf[5] = 11

第二次 ts_init_sec(5, seq=21):
  读 flash header → magic=TSDB, erase_cnt=11
  old_ec = max(ec_buf[5]=11, flash=11) = 11
  erase → write header(ec=12)
  ec_buf[5] = 12

  如果不取 max 而是直接赋值:
  old_ec = flash=11 → 结果相同 (此场景无差异)

差异场景: ec_buf 被外部修改或初始化错误
  ec_buf[5] = 15 (被其他逻辑设置)
  flash header erase_cnt = 11
  取 max: old_ec = max(15, 11) = 15 → new ec = 16 ✓
  不取 max: old_ec = 11 → new ec = 12 → 回退! ✗
```

### 验证点

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 正常连续调用 | ✓ | ec 单调递增 |
| RAM > Flash 场景 | ✓ | 取 max 防止回退 |

## 测试用例 6: seq 非单调初始化

### 场景构造

Flash 中三个 sector 的 seq 非单调（可能由多次掉电导致）：

```
Sector 0: SEALED, seq=5
Sector 1: SEALED, seq=7
Sector 2: SEALED, seq=6  ← 非单调
Sector 3..7: EMPTY
```

### init 行为追踪

```
Phase 1:
  Sector 0: SEALED, seq=5 → head候选(seq=5), tail候选(seq=5)
  Sector 1: SEALED, seq=7 → head=1(seq=7)
  Sector 2: SEALED, seq=6 → 不影响 head(7>6), tail仍是0(5<6)
  Sector 3..7: EMPTY
  
  head_s=1(seq=7), tail_s=0(seq=5)

Phase 2 连续性检查 (tail=0 → head=1):
  s=0: SEALED, seq=5, first=true → prev_seq=5, first=false
  s=1: SEALED, seq=7
    RDB_SEQ16_GT(7, 5) → true → OK
  s=1 == head → break

  等等，sector 2 在 tail→head 路径上吗?
  tail=0, head=1: 路径 0→1 (直接)
  sector 2 不在 0→1 路径上 → 不参与连续性检查
  
  但实际的 ring 应该包含所有有数据的 sector...
  
  重新分析: tail_sq=5(sector 0), head_sq=7(sector 1)
  遍历从 tail(0) 到 head(1):
    s=0 → s=1: 只经过 sector 0 和 1
  sector 2 的 seq=6 被忽略

  这意味着 sector 2 中的数据在 query 时仍可能被访问到
  (如果 from tail 遍历经过它)，但 init 不将其纳入环形。
  
  实际上 tail→head 的遍历是:
    s=tail=0, 循环 i<sector_cnt(8):
      i=0: s=0, 处理, 0≠head(1), s=next(0)=1
      i=1: s=1, 处理, 1==head → break
  所以 sector 2 不在 tail→head 路径上，不被检查。

  但在 Phase 4 total_count 中:
    s=tail=0:
      0≠head(1) → ts_sector_count(0) → 比如 100
      s=next(0)=1: 1==head → recount += head_count
      break
    sector 2 不被计数 → total_count 不包含 sector 2 的记录

  在 query 中:
    s=tail=0:
      处理 sector 0
      0≠head(1), s=1
    s=1:
      处理 sector 1 (head)
      1==head → break
    sector 2 不被查询 → 数据丢失!
```

**发现问题**: 当存在非单调 seq 时，seq 介于 tail_sq 和 head_sq 之间的"孤立"扇区会被排除在 tail→head 遍历路径之外，导致其中数据无法被查询或计数。

**评估**: 这是 data_gaps 降级策略的已知代价。非单调 seq 本身是异常状态（多次掉电导致），丢失部分异常扇区的数据是可接受的折中，优于直接 format 丢失全部数据。stats.data_gaps > 0 可告知应用层此异常的存在。

### 验证点

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 不触发 format | ✓ | data_gaps++ 替代 |
| head/tail 正确识别 | ✓ | seq 最大/最小 |
| 孤立扇区数据不可达 | 预期行为 | stats.data_gaps 告警 |
