### 2.5 [BUG-05] TSDB Format Rejects NULL erase_cnts — CRITICAL REGRESSION

**File**: `src/rocketdb_tsdb.c:1024`

```c
if (!db->erase_cnts)
    return RDB_ERR_PARAM;
```

This check was added in a previous review fix for consistency with KVDB. However, it introduces a regression:

1. **API contract violation**: `rdb_tsdb_init()` at `rocketdb.h:1017` documents that `ec_buf` "May be NULL if erase count tracking is not needed."
2. **Init→format path broken**: When `rdb_tsdb_init()` discovers a blank flash (no data sectors), it falls through to `rdb_tsdb_format(db)` at line 780. If the caller passed NULL for `ec_buf`, format now rejects it with `RDB_ERR_PARAM`.
3. **KVDB doesn't have this issue**: KVDB's `sectors` array is mandatory (sector metadata is required for operation). TSDB's `erase_cnts` is optional — it's only needed for wear-leveling statistics.

**Root cause**: The TSDB `erase_cnts` array is semantically different from KVDB's `sectors` array. KVDB requires `sectors` for runtime metadata. TSDB only uses `erase_cnts` for wear tracking; the engine functions correctly without it (all code paths already handle NULL via `db->erase_cnts ? db->erase_cnts[s] : 0`).

**Fix**: Remove the NULL check for `erase_cnts` in `rdb_tsdb_format`, or allow NULL when `sector_cnt == 0` (not pre-set by caller). Keep the `sector_cnt` consistency check.

### 2.6 [BUG-06] KVDB Format Missing Min Sector Count Validation — LOW

**File**: `src/rocketdb_kvdb.c:2041-2043`

```c
{
    uint32_t scnt32 = db->part->total_size / db->part->sector_size;
    if (scnt32 == 0 || scnt32 > RDB_MAX_SECTORS)
        return RDB_ERR_PARAM;
}
```

Missing `scnt32 < RDB_KV_MIN_SECTORS` check. Compare with `rdb_kvdb_init` at line 1846:
```c
if (scnt < RDB_KV_MIN_SECTORS || scnt > RDB_MAX_SECTORS)
    return RDB_ERR_PARAM;
```

A format with 1-2 sectors would succeed, but subsequent `rdb_kvdb_init` would fail. Inconsistency between format and init validation.

**Fix**: Add `|| scnt32 < RDB_KV_MIN_SECTORS` to the condition.

## 3. Improvements

### 3.1 [IMP-01] Add TSDB Init NULL erase_cnts Test — MEDIUM

No test covers the `ec_buf = NULL` path in `rdb_tsdb_init()`. The existing tests always pass a valid `g_ec` array. A test with NULL `erase_cnts` would have caught BUG-05.

### 3.2 [IMP-02] W25QXX Command Name Typo in Template — LOW

**File**: `interface/rocketdb_interface_template.c:45`

```c
 *   #define W25Q_CMD_WRLR        0x04  // Write Disable
```

The W25QXX command for Write Disable is `WRDI` (0x04), not `WRLR`. This is a typo in the reference comment.

### 3.3 [IMP-03] TSDB Query RDB_TRUE/RDB_FALSE Semantics Defect — LOW

The constants `RDB_TRUE` (1) and `RDB_FALSE` (0) are defined in `rocketdb.h` for boolean returns. But they are also used as the `recover` parameter in `ts_scan()` and as an `update_woff` parameter in `scan_sector()`. This conflates "boolean" with "write behavior" semantics — a future refactor should use a dedicated enum or named constants (e.g., `TS_SCAN_READONLY` / `TS_SCAN_RECOVER`) for code clarity.
